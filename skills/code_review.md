请生成一个 code review 计划，对 src 下面的文件、模块、进行 code review
- 写法现代，用 rust 最新发行版的写法
- 时间复杂度和空间复杂度尽量最优化
   - 尽量用流、迭代器按需生成，让空间复杂度是 O(1) 而不是 O(n)
- 对性能敏感的场景，优先考虑高效算法与数据结构。
   - 多用迭代器，少于 for，可以减少越界检查
   - 对于可以提前终止的，要避免全遍历（find、scan、take_while、try_for_each+ControlFlow），能不 collect，就避免 collect
- 尽可能消除异步的闭包捕获，手动精简捕获的变量，只捕获必要的字段，而不是整个 &self
- 对于冷路径可以用#[cold] 标注（忌乱加）
- 大胆用 get_unchecked、unwrap_unchecked、在 100%不会出现问题的地方减少开销和越界检查
- 模块低耦合、高内聚。拆分过于庞大的文件为模块文件夹，拆分过于庞大的函数为几个模块文件
- 暴露的接口清晰优雅，易于使用和维护。
- 对于模块内部用的结构体，直接曝光字段，而不是写包装函数
- 避免重复代码或雷同类：
  - 可通过函数抽象、泛型参数或泛型 trait 减少冗余
- 兼顾性能与设计，避免过度设计

- 确保代码无潜在 BUG
- 代码必须符合生产环境要求和行业最佳实践
小步迭代，每当完成一个修改，都运行../sh/clippy.sh， 修复警告（避免直接使用 allow，除非绝对必要）， 然后运行 ./test.sh （如果修改了代码，检查是否有相关的测试。没有请补全，如果有，不要重复写。尽量在 tests 目录下面用单独文件写测试，除非是测试一些内部属性什么的，这种情况，请创建一个 test.rs 写进程测试，不写成文档测试）
